# Roadmap

## Principles

- Preserve behavior first.
- Refactor in small vertical slices.
- Prefer one use case per extraction.
- Keep parser internals away from UI-facing components.
- Keep feature docs active and concise; compress completed refactor history
  into this roadmap or `docs/specs/plans.md`.

## Completed Slices

1. Extract core use cases:
   build unit list, build unit list view, export unit list CSV, normalize AJS
   document, build flow graph, show unit definition, editor feedback, and
   telemetry.
2. Stabilize application-facing DTOs so table, flow, CSV, diagnostics, hover,
   and show-unit-definition behavior no longer depend directly on parser or
   wrapper internals.
3. Consolidate wrapper-derived semantics selectively:
   shared wait-state, priority, relation, layout, recovery, root-jobnet,
   schedule, depth, encoded-string, and normalization rules are shared where
   reuse is real; unit-local JP1/AJS behavior remains on the owning wrappers.
4. Decompose high-risk internal files:
   `ParameterFactory.ts`, `buildUnitListView.ts`, and duplicated wrapper
   boilerplate now have focused helper structure and regression coverage.
5. Reduce activation, bootstrap, webview, telemetry, diagnostics, hover, and
   viewer wiring into clearer runtime and adapter seams.
6. Modernize runtime boundaries:
   remove stale `flatted` assumptions, migrate to pinned `pnpm`, split table
   and flow webview bundles, and document bundle-size evidence.
7. Improve flow-view usability:
   JP1/AJS View-inspired visual refresh, nested expansion, current-scope
   search, expand-all, and list/flow bridge navigation are in place.
8. Start JP1/AJS v13 command-generation alignment:
   `ajsshow` and `ajsprint` are generated by an application helper and exposed
   through a localized command-builder surface.
9. Extend list-view search:
   table global search remains presentation-local while matching rendered row
   values plus normalized parameter values in value mode, or normalized
   `key=value` candidates in key-value mode.

## Current Roadmap

1. Re-base parameter interpretation on JP1/Automatic Job Management System 3
   version 13 Definition File Reference, starting from the documented audit of
   current shared parameter-semantics seams.

   - Add traceability from JP1/AJS v13 manual sections to supported parameter
     keys, parser behavior, normalized model fields, use cases, and regression
     tests.
   - Start with high-impact parameters already consumed by unit list, flow graph,
     hover, diagnostics, and unit definition rendering.
   - Keep behavior-preserving slices separate from behavior-changing manual
     alignment slices.

2. Add a read-only JP1/AJS WebAPI import path for loading server-side
   definition data.

   - Keep transport, authentication, and endpoint details in infrastructure.
   - Expose imported definitions through application-facing DTOs or use cases.
   - Reuse the normalized AJS model where imported data represents the same
     product concept as parsed definition files.
   - Keep WebAPI import read-only until the initial boundary and tests are
     stable.
   - Start with a desktop extension command, a structured application import
     port, and an unsupported-host result for browser execution until a
     browser-safe adapter is explicitly designed and tested.
   - Follow JP1 Version 13 JP1/Automatic Job Management System 3 Command
     Reference, manual 3021-3-L49-20(E), Part 3 API, and record section-level
     traceability for the first endpoint implemented.
   - Define supported endpoints in a repository-local OpenAPI contract derived
     from the manual, then generate mock servers and infrastructure/test stubs
     from that contract to keep integration tests stable. Keep OpenAPI source
     files under `docs/specs/features/import-definition-via-webapi/openapi/`.
   - Offer the feature as beta until smoke verification against a real JP1/AJS3
     environment is recorded.

3. Replace the custom `UnitEntity` hash implementation with a common algorithm
   only after identity and compatibility checks are refreshed.

   - Treat hash behavior as an implementation detail unless persisted or
     user-visible identity semantics are affected.
   - Add compatibility fixtures before changing the algorithm.

4. Consolidate i18n translation files where duplication remains high enough to
   justify the change.

   - Keep this as a targeted cleanup, not a broad localization rewrite.

5. Strengthen normalized-model convergence.

   - Reduce remaining raw `Unit` and wrapper-oriented dependencies in
     application-facing paths where a stable `AjsDocument` / `AjsUnit` contract
     already exists.
   - Keep unit-local JP1/AJS behavior on wrappers when it is not reused across
     consumers.
   - Promote only cross-consumer semantics into normalized helpers, such as
     dependency lookup, inheritance, root-jobnet defaults, wait-state behavior,
     and parameter interpretation rules.

6. Introduce stricter parser/infrastructure boundaries.

   - Define an application-facing parser or document-loading port.
   - Move concrete parser orchestration behind an adapter boundary when
     practical.
   - Keep generated ANTLR artifacts and parser-specific mechanics out of
     application use cases.
   - Preserve current desktop and web extension behavior while migrating.

7. Use Qlty findings as architectural feedback.

- Treat recurring duplication, complexity, and nested-control-flow findings
  as prioritized refactoring candidates.
- Correlate Qlty findings with SDD tasks and technical debt slices.
- Use Qlty trends to guide decomposition work rather than broad cleanup.

## Deferred / Optional Slices

1. Introduce a shared search use case only if list, flow, or another non-table
   consumer needs common query semantics. Until then, keep list search
   presentation-local.
2. Extend JP1/AJS WebAPI support beyond read-only import only after the initial
   boundary and authentication model are stable.
3. Revisit viewer-specific bundle-size reductions only if a future
   compatibility, startup, or payload target creates stronger pressure than
   current marginal gains justify.
4. Add deeper JP1/AJS View interaction parity only after the current visual
   refresh and nested expansion behavior settle.
5. Revisit directory structure under `src/extension/webview/` only if the
   remaining files stop reading as one cohesive viewer module.

## Done Criteria For A Slice

- Use case or feature requirement is documented when behavior or boundary
  decisions change.
- Affected boundaries are named explicitly.
- Tests cover the behavior being preserved or changed.
- Desktop and web compatibility impact is stated.
- Remaining debt is listed only when it is actionable.
- Feature `TASKS.md`, branch plans, and roadmap are updated in the same commit
  when priorities or repository sequencing change.
- JP1/AJS manual-dependent work names the target product version and source
  document.
